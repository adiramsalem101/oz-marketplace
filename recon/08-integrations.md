## Notes on the AI Import discrepancy (row 6 vs. row 7)

The docs (`ONBOARDING_DAGAN.md:85`, `B2B_SPECS_MVP.md:241, 253`, `GLOSSARY.md:79`) all say AI Import uses **Anthropic Haiku**.
The actual legacy code at `oz-marketplace-legacy/supabase/functions/ai-import/index.ts:1` imports `npm:@google/generative-ai`. This is **Google Gemini**, not Anthropic.

This is the single largest doc↔code drift uncovered by the recon. The rebuild must either (a) re-implement AI Import on Anthropic (matching docs), or (b) update docs to acknowledge Gemini. Decision required from Adir.

## Notes on the Stripe presence in `hostel_bookings`

`hostel_bookings.stripe_session_id` column lives in the production schema (`baseline.sql:524`) despite IRON_RULE 2 banning Stripe in production. The column was introduced by `_archive/016_hostel_bookings.sql` and `_archive/017_run_in_prod.sql`. Likely artifact of a hostel-pilot Stripe Test integration that pre-dated the IRON_RULE.

Per IRON_RULE 1 (schema isolation), the hostel domain has its own rules — but the explicit IRON_RULE 2 ban applies to **all of production**, including the hostel domain. The rebuild plan should call out:
- Option A: drop the column (clean break, requires data migration plan if hostel pilot data must persist).
- Option B: rename to `external_payment_session_id` and reuse for the future Pelecard integration.

Flagged in `10-open-questions.md`.
